# IMU 数据采集计划

> 版本：v0.2（面向耳机 IMU 的姿势提醒与基础动作识别）  
> 现场执行、文件命名、Schema v2 标签和批次验收以
> [`docs/IMU_DATA_COLLECTION_AND_LABELING_STANDARD.md`](docs/IMU_DATA_COLLECTION_AND_LABELING_STANDARD.md)
> 为准；如与本计划冲突，以该规范为准。
> 本计划将“耳机可直接观测的头部姿态”与“需要额外参考才能判断的完整坐姿”分开处理。首期不以 IMU 单独诊断脊柱、肩部或腰背姿势。
>
> **【标注说明】带“新增/调整”标识的模块或条目，为本次补充内容。**

## 1. 采集目标与范围

### 1.1 首期目标（P0） **【新增/调整】**

建立可验证耳机 IMU 数据链路与算法能力，优先支持：

1. 头部相对个人基线的低头/抬头、左/右侧倾、左/右转头；
2. 持续低头、持续偏头、长时间姿态少变化等提醒事件；
3. 基础活动状态：静坐、站立、步行、坐下、起立；
4. 识别会造成误触发的日常行为：说话、咀嚼、喝水、看手机、戴取耳机、耳机松动。

### 1.2 次期目标（P1） **【新增】**

- 基础运动识别：深蹲、弓步蹲、开合跳、俯卧撑、平板支撑；
- 探索头部姿态与躯干姿态的相关性。

> 注意：单耳机 IMU 不能直接测量圆肩、耸肩、胸椎后凸、骨盆后倾、腰椎曲度或翘腿。若后续要训练“完整坐姿”模型，须增加躯干 IMU、坐垫压力或同步视频/动作捕捉参考。

### 1.3 不纳入本期结论 **【新增】**

- 医疗诊断、颈椎病/腰痛风险判断；
- 头部空间位置、离屏距离；
- 未经额外传感器验证的肩、腰、骨盆姿势。

## 2. 受试者与样本策略 **【调整】**

| 人群 | 定义 | Pilot 建议人数 | 建模建议 |
| --- | --- | ---: | ---: |
| 日常人群 | 无固定健身习惯，日常活动水平 | 10 | 20+ |
| 健身入门 | 每周运动 1–2 次 | 10 | 20+ |
| 长期训练者 | 长期规律训练，动作控制较好 | 10 | 20+ |

- 第一阶段 30 人可用于**链路、标签和可行性验证**，不应宣称已覆盖泛化人群。
- 建模评估必须按 `subject_id` 划分训练/验证/测试集；同一人的不同组数据不可同时进入训练和测试。
- 尽量平衡性别、身高、头围/耳廓、是否佩戴眼镜、发型、惯用手和年龄段；这些信息用于分析偏差，不用于身份识别。
- 有急性疼痛、眩晕、近期手术、严重平衡问题或不适合运动者，不参加高强度动作采集。

## 3. 动作与场景设计

### A. P0：头颈静态姿态与桌面场景 **【新增】**

每个静态姿态建议 30 秒 × 3 组；姿态之间回到自然中立位 10 秒。

| 类别 | 动作/场景 | 标签建议 |
| --- | --- | --- |
| 基线 | 自然坐姿、自然站姿 | `neutral_sitting`、`neutral_standing` |
| 俯仰 | 低头 10°/20°/30°、抬头 | `head_pitch_down/up` + 角度档位 |
| 侧倾 | 左/右歪头 | `head_roll_left/right` + 角度档位 |
| 转头 | 左/右转头 | `head_yaw_left/right` + 角度档位 |
| 组合姿态 | 低头+左/右转头；低头+左/右侧倾 | 多标签 |
| 桌面任务 | 正视屏幕打字、看笔记本键盘、看手机、看副屏、阅读纸质材料 | `desk_task` + 姿态标签 |
| 持续暴露 | 持续低头/偏头 1、3、5 分钟 | `sustained_exposure` + 时长 |

**重要补充**：对“前伸头”单独标注为 `forward_head_candidate`，并同步拍摄侧面参考视频或使用躯干参考。耳机只测头部方向，不能仅凭单耳机 IMU把它当作临床前伸头真值。

### B. P0：姿态转换与基础活动

| 动作 | 每组次数或时长 | 组数 |
| --- | ---: | ---: |
| 坐下 → 站起 | 10 次 | 3 |
| 站立 → 坐下 | 10 次 | 3 |
| 静坐、站立 | 30 秒 | 3 |
| 正常步行、快走 | 30 秒 | 3 |
| 上/下楼梯 | 1 层或 30 秒 | 3 |
| 弯腰取物 | 10 次 | 3 |

### C. P0：真实佩戴与误触发场景（必须采集） **【新增】**

| 场景 | 建议时长/次数 | 目的 |
| --- | ---: | --- |
| 正常说话、电话会议 | 各 60 秒 | 评估语音/下颌运动干扰 |
| 咀嚼、喝水 | 各 30 秒或 10 次 | 评估下颌与头部短时运动误报 |
| 戴上、取下、重新佩戴 | 10 次 | 识别佩戴事件与重校准需求 |
| 浅戴/深戴、左右不对称、偏松 | 每种 30 秒 | 评估佩戴误差 |
| 单耳佩戴、双耳佩戴 | 各 30 秒 | 评估产品支持边界 |
| 播放音乐、ANC、通话、空间音频（如支持） | 每种 60 秒 | 评估系统模式干扰 |
| 走路时低头看手机、转头交谈 | 各 60 秒 | 评估动态误触发 |

### D. P1：健身动作

| 动作 | 每组次数或时长 | 组数 |
| --- | ---: | ---: |
| 深蹲 | 10 次 | 3 |
| 弓步蹲 | 左右各 10 次 | 3 |
| 开合跳 | 30 秒 | 3 |
| 俯卧撑 | 10 次 | 3 |
| 平板支撑 | 30 秒 | 3 |
| 卷腹 | 10 次 | 3 |

> 如时间有限，先完成 A、B、C；它们与耳机姿势提醒的产品价值直接相关。健身动作宜另开采集批次，避免单人连续采集过长导致疲劳、动作变形和标签失真。

## 4. 真值、标签与同步

### 4.1 标签原则 **【新增】**

- 每条 trial 必须有 `action_name`、`start_time`、`end_time`、`label_source`、`label_confidence`。
- 动作起止不可仅靠人工回忆；优先采用口令/按钮/拍手等同步事件，并在视频中可见或可听。
- 连续动作按帧或按时间段标注；重复动作额外记录 `repetition_id`。
- `quality_label` 不能只写“标准/异常”，应注明判定人、判定规则和置信度。
- 至少抽取 10% 数据由第二名标注者复核，记录标签一致性和冲突处理结果。

### 4.2 参考真值层级 **【新增】**

| 目标 | 最低可用真值 | 推荐真值 |
| --- | --- | --- |
| 低头/歪头/转头 | 采集员口令 + 同步视频 | 侧/前方视频 + 角度板或光学动捕 |
| 坐下/起立/步行 | 视频或人工事件标记 | 视频 + 压力垫/脚部参考 |
| 前伸头/完整坐姿 | 不建议仅人工观察 | 侧面视频 + C7/肩部标记，或躯干 IMU |
| 健身动作质量 | 教练人工评分 | 同步视频 + 教练评分规则 |

### 4.3 时间同步要求 **【新增】**

- 使用单调递增的设备时间戳，不使用手机墙钟作为唯一时间依据；记录单位与时钟来源。
- 保存 `sequence_id`、采样时钟、丢包数、重连事件、蓝牙延迟估计值。
- 若采集视频，开始和结束各放置一次同步事件；记录视频帧率、起始时间和同步偏差。
- 左右耳数据必须包含 `ear_side` 和各自时间戳；不可假设两侧天然同步。

## 5. 必采数据字段

### 5.1 原始传感器与运行状态 **【调整】**

| 字段 | 必须 | 说明 |
| --- | --- | --- |
| `timestamp_monotonic_ns` | 是 | 单调递增高精度时间戳 |
| `sequence_id` | 是 | 检查丢包、重排 |
| `ear_side` | 是 | `left` / `right` |
| `acc_x/y/z`、`gyro_x/y/z` | 是 | 原始值或已标定值；必须注明单位 |
| `mag_x/y/z` | 设备支持时 | 同时记录磁场质量/是否参与融合 |
| `temperature` | 推荐 | 分析温漂 |
| `sample_rate_configured/observed` | 是 | 配置值与实测值都保存 |
| `imu_range`、`filter_config` | 是 | 量程、滤波、ODR、融合版本 |
| `sensor_calibration_state` | 推荐 | 出厂/运行时校准状态 |
| `battery_level`、`charging_state` | 推荐 | 分析续航与异常 |
| `firmware_version`、`app_version` | 是 | 数据可追溯 |
| `audio_mode`、`anc_mode`、`call_state` | 推荐 | 排查系统模式干扰 |

### 5.2 受试者、试次和环境 **【调整】**

保留：`subject_id`、年龄段、身高、体重、运动习惯、动作限制、惯用手。增加：

- `session_id`、`trial_id`、`protocol_version`、`collector_id`；
- `wearing_position`、耳塞尺寸、佩戴深浅、松紧、是否眼镜/帽子；
- 环境：室内/室外、桌面材质、是否靠近笔记本/金属/磁吸配件；
- `recalibration_event`、`device_reinsert_event`、`abnormal_event`；
- `consent_version` 和去标识化状态。

### 5.3 单位与坐标系（必须写入数据字典） **【新增】**

- 加速度：`m/s²` 或 `g`；角速度：`°/s` 或 `rad/s`；磁力：`µT`；不可混用。
- 明确设备坐标系、左右耳坐标轴镜像关系、正方向定义。
- 明确传感器数据是原始数据、工厂校准数据，还是已做姿态融合后的数据。

## 6. 推荐采样策略 **【调整】**

| 用途 | 推荐 ODR | 说明 |
| --- | ---: | --- |
| 静态头颈姿态/久坐提醒 | 25–50 Hz | 首期姿势提醒的主目标 |
| 坐站转换、步行 | 50–100 Hz | 兼顾事件边界与功耗 |
| 跳跃、开合跳等快速动作 | 100 Hz 及以上 | 以实际芯片带宽和续航为准 |

- 不建议为了“方便”一律 100 Hz；需先确认耳机真实可持续输出的 ODR、传输稳定性与续航。
- 采集时记录实际到达率和时间间隔分布，不能只记录配置采样率。
- 若最终产品只运行 25/50 Hz，训练集必须包含相同 ODR 或可复现的降采样版本。

## 7. 单人采集流程 **【调整】**

1. 签署同意、登记去标识化受试者信息，确认无不适合动作采集的情况；
2. 记录设备 ID、固件/App 版本、IMU 配置、耳塞尺寸和左右耳佩戴状态；
3. 采集 30 秒静止原始噪声基线（桌面放置 + 正常佩戴）；
4. 完成自然中立位校准，并记录校准姿态与校准时间；
5. 采集 P0 静态头颈/桌面场景；
6. 采集 P0 姿态转换、基础活动、误触发场景；
7. 如参与 P1，再采集健身动作；
8. 现场自动检查：时间戳连续性、丢包、实际采样率、数据量程饱和、左右耳同步、视频同步；
9. 记录异常、疲劳、耳机滑动、重戴、动作中断和主观不适；
10. 导出后生成数据完整性报告，再允许受试者离场。

建议拆分：P0 每人 35–50 分钟；P1 健身动作另设 20–30 分钟会话。原计划的“所有类别 20–30 分钟完成”会导致疲劳、动作质量下降和漏标。

## 8. 数据质量验收 **【新增】**

每个 trial 自动生成质检结果，至少检查：

- 时间戳单调递增，间隔异常比例；
- `sequence_id` 丢失/重排数量；
- 实际采样率均值、标准差、最小/最大间隔；
- 加速度/陀螺仪是否饱和、全零、恒定值或异常尖峰；
- 左右耳是否都有数据、同步偏差是否在设定阈值内；
- 动作标签是否完整、与视频/同步事件是否对齐；
- 固件、IMU 配置、坐标系、单位是否齐全；
- 佩戴异常、重连、重校准、磁扰等是否被记录。

建议设置三种状态：`accepted`、`accepted_with_note`、`recollect_required`，禁止只凭“文件存在”视作合格。

## 10. 必须与硬件工程师确认的事项 **【新增】**

以下问题必须在正式采集前闭环；没有答案时，不应开始大规模采集。

| 需要硬件判断的问题 | 为什么影响采集/算法 | 需要的明确输出 |
| --- | --- | --- |
| 现有耳机是否有 IMU，具体芯片与 6/9 轴配置？ | 决定可采字段与姿态能力边界 | 芯片型号、数据手册、左右耳配置 |
| 能否访问原始 `acc/gyro/(mag)` 数据？ | 仅有系统头追数据会限制算法迭代 | API/固件接口、单位、量程、精度 |
| 实际可持续 ODR、FIFO、时间戳精度与丢包行为？ | 决定 25/50/100 Hz 是否可行 | 实测采样间隔、延迟、丢包率报告 |
| 左右耳是否同步，坐标系是否镜像？ | 影响双耳融合和标签解释 | 坐标系文档、时间同步方案/误差 |
| IMU 在耳机壳体中的位置与装配公差？ | 影响头部姿态映射与重复佩戴误差 | 安装位置、轴向、装配图、误差估计 |
| 耳塞尺寸/耳型/松动/重戴导致的误差有多大？ | 决定是否每次会话校准 | 重复佩戴测试方案与误差分布 |
| 磁力计是否存在，是否受扬声器、磁铁、充电线圈影响？ | 9 轴 Yaw 可能在室内失真 | 磁扰测试、融合启停策略 |
| 音乐、ANC、通话、空间音频是否占用 IMU 或改变数据质量？ | 决定真实使用时能否持续监测 | 各模式下数据可用性/噪声/功耗 |
| 持续采样、端侧处理、蓝牙传输分别消耗多少电？ | 决定产品续航与采样策略 | 25/50/100 Hz 功耗预算与续航影响 |
| 姿态融合在传感器、耳机 MCU、手机还是系统 SDK？ | 决定可控性、版本追溯和算法迭代 | 数据链路图、可输出字段、版本机制 |
| 是否可输出佩戴状态、温度、校准状态、重连事件？ | 用于排除异常样本与提升鲁棒性 | 可用状态字段清单 |
| iOS/Android 锁屏、后台、通话时是否能持续取数？ | 决定产品是否能真实落地 | 双平台限制、后台时长、权限要求 |

### 10.1 与硬件会议必须带走的结论 **【新增】**

1. 首期可拿到的数据字段、单位、坐标系和最高稳定采样率；
2. 是否能同时支持音频功能与 IMU 连续采集；
3. 开启姿势监测后对续航的实测或估算影响；
4. 重新佩戴、左右耳、不同耳塞条件下的姿态重复性；
5. 是否采用 6 轴为首期主方案，以及 9 轴磁力计的使用/禁用条件；
6. 原始数据、姿态数据、事件数据分别在哪里计算、缓存、传输和保存；
7. 可在何时拿到用于 Pilot 的固件、日志工具和数据格式定义。

## 11. 最小可行采集版本（建议先执行） **【新增】**

在硬件数据链路未完全稳定前，先用 6–10 名受试者完成 Pilot：

1. 30 秒静止基线；
2. 自然坐姿、低头、歪头、左/右转头及其持续 1 分钟版本；
3. 正视屏幕、看键盘、看手机、看副屏；
4. 坐站转换、步行；
5. 说话、咀嚼、喝水、重戴、偏松佩戴；
6. 同步参考视频和完整元数据；
7. 输出数据质量报告与重复佩戴误差报告。

Pilot 通过后，再扩大到 30 人和 P1 健身动作，避免在错误的采样率、时间同步或坐标系定义上积累大量不可用数据。
